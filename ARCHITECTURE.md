## ğŸ” Trust Model

### What is Trusted

| Component | Trust Level | Rationale |
|-----------|-------------|-----------|
| Ethereum Network | High | Consensus mechanism ensures integrity |
| Smart Contract Code | High | Immutable once deployed, verified on Etherscan |
| MetaMask Signatures | High | Cryptographic proof of user intent |
| IPFS Content | Medium | Content-addressed (CID = hash), but availability depends on pinning |

### What is Not Trusted

| Component | Mitigation |
|-----------|------------|
| User Input | Validated on-chain before state changes |
| Frontend Code | Can be verified, but contract is source of truth |
| IPFS Gateways | Multiple gateway fallbacks implemented |
| RPC Provider | Standard Ethereum API, can switch providers |

## ğŸ¯ Key Design Decisions

### 1. On-chain Escrow Instead of Direct Transfer

**Decision**: Funds are held by the smart contract during escrow period.

**Rationale**:
- Provides buyer protection without third-party arbitration
- Automatic release after deadline reduces support burden
- Contract is neutral arbiter

**Trade-off**: Funds locked for up to 14 days if buyer doesn't confirm.

### 2. IPFS for Metadata, Not On-chain Storage

**Decision**: Store only the metadata URI (CID) on-chain.

**Rationale**:
- Storing images on-chain would cost thousands of dollars in gas
- IPFS provides content-addressed verification
- Metadata is immutable once pinned

**Trade-off**: Depends on pinning service availability.

### 3. Server-side IPFS Uploads

**Decision**: IPFS uploads go through `/api/ipfs/upload` route, not client-side.

**Rationale**:
- Protects Pinata JWT from exposure in browser
- Allows server-side validation before pinning
- Consistent error handling

**Trade-off**: Additional latency through server.

### 4. Pull Pattern for Bid Withdrawals

**Decision**: Outbid users must call `withdrawBid()` to retrieve funds.

**Rationale**:
- Avoids reentrancy vulnerabilities with push payments
- Standard secure pattern (OpenZeppelin recommendation)
- Gas cost borne by recipient, not contract

**Trade-off**: Requires user action to retrieve funds.

### 5. Single Contract Architecture

**Decision**: All marketplace logic in one contract.

**Rationale**:
- Simpler deployment and upgrades
- Lower gas costs (no cross-contract calls)
- Easier to reason about state

**Trade-off**: Larger contract size, less modularity.
